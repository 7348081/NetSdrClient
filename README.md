ЛАБОРАТОРНА №1 Підключення SonarCloud і CI 

Метою першої лабораторної роботи було підключення системи автоматичного аналізу якості коду SonarCloud до GitHub-репозиторію та налаштування безперервної інтеграції за допомогою GitHub Actions. 

На першому етапі було створено проєкт у сервісі SonarCloud і прив’язано його до публічного репозиторію GitHub. Далі було згенеровано спеціальний токен доступу, який дозволяє GitHub Actions передавати результати аналізу в SonarCloud. Цей токен додається в репозиторій як секретна змінна. 

Після цього було створено workflow-файл GitHub Actions, який автоматично запускається при кожному push або pull request у головну гілку. Workflow виконує завантаження коду, встановлення .NET SDK, збірку проєкту та запуск SonarScanner — інструмента, який аналізує код і відправляє звіт у SonarCloud. 

У результаті лабораторної роботи отримано автоматизований процес перевірки якості коду. Кожна зміна в репозиторії перевіряється без участі розробника, що є основою сучасної практики DevOps і реінжинірингу програмного забезпечення. 


ЛАБОРАТОРНА №2 Виявлення та усунення Code Smells 

Метою другої лабораторної роботи було навчитися знаходити та виправляти так звані Code Smells у програмному коді. 

Code Smells — це ознаки неякісної структури коду, які не є прямими помилками, але ускладнюють розуміння, підтримку і розвиток програми. До типових прикладів належать занадто довгі методи, дублювання логіки, надлишкові залежності та складні умовні конструкції. 

У лабораторній роботі за допомогою SonarCloud було виконано аналіз репозиторію та отримано список знайдених проблем. Далі ці проблеми усувалися шляхом рефакторингу — зміни структури коду без зміни його функціональності. 

Таким чином, лабораторна №2 демонструє базовий підхід реінжинірингу: покращення якості існуючого коду без переписування системи з нуля. 

  

ЛАБОРАТОРНА №3 Unit Tests та Test Coverage 

Метою третьої лабораторної роботи було додавання модульних тестів до проєкту та підвищення рівня покриття коду тестами. 

Unit Tests — це автоматичні тести, які перевіряють роботу окремих функцій або класів. Вони дозволяють швидко виявляти помилки в логіці та гарантують стабільність роботи коду після внесення змін. 

Coverage або покриття тестами — це метрика, яка показує, який відсоток коду виконується під час запуску тестів. Високе покриття означає, що більшість логіки перевірено автоматично. 

У межах лабораторної було створено тестовий проєкт, написано базові модульні тести та налаштовано збір метрик coverage. Отримані результати передаються в SonarCloud, що дозволяє контролювати рівень тестового покриття в Quality Gate. 

  

ЛАБОРАТОРНА №4 Пошук і усунення дублювання коду 

Метою четвертої лабораторної роботи було виявлення та усунення дубльованих фрагментів коду. 

Code Duplication — це повторення однакової або дуже схожої логіки в різних частинах проєкту. Наявність дублікатів ускладнює підтримку, оскільки при зміні логіки необхідно редагувати кілька місць одночасно. 

SonarCloud автоматично знаходить дублікати та відображає їх у звіті. Далі виконувалося винесення спільної логіки в окремі методи або класи. 

У результаті лабораторної роботи зменшено повторення коду та покращено його структурованість, що є важливою частиною реінжинірингу. 

  

ЛАБОРАТОРНА №5 Архітектурні правила і NetArchTest 

Метою п’ятої лабораторної роботи було введення архітектурних правил і перевірка правильності залежностей між модулями проєкту. 

Архітектура програмної системи визначає, як її компоненти взаємодіють між собою. Порушення архітектурних правил призводить до хаотичних залежностей і ускладнює розвиток системи. 

У лабораторній використовується бібліотека NetArchTest, яка дозволяє автоматично перевіряти архітектурні обмеження. Наприклад, можна задати правило, що один модуль не повинен напряму залежати від іншого. 

Таким чином, архітектурний контроль автоматизується і стає частиною CI-процесу. 

  

ЛАБОРАТОРНА №6 Рефакторинг для покращення тестованості 

Метою шостої лабораторної роботи було виконання безпечного рефакторингу для підвищення тестованості коду. 

Рефакторинг — це процес зміни внутрішньої структури коду без зміни зовнішньої поведінки. Його метою є спрощення логіки, зменшення зв’язності компонентів та покращення читабельності. 

У лабораторній виконувалося виділення інтерфейсів, розділення відповідальностей між класами та усунення жорстких залежностей. Це дозволило легше писати модульні тести. 

Отже, лабораторна №6 демонструє, як реінжиніринг покращує не тільки структуру коду, а й можливість його автоматичного тестування. 

  

ЛАБОРАТОРНА №7 Оновлення залежностей і безпека 

Метою сьомої лабораторної роботи було оновлення застарілих бібліотек та перевірка залежностей на наявність вразливостей. 

Будь-який сучасний проєкт використовує зовнішні пакети. Якщо ці пакети застаріли або містять відомі вразливості, це створює ризик для безпеки всієї системи. 

У лабораторній використано інструменти GitHub Dependabot та аналіз SonarCloud для виявлення небезпечних або застарілих залежностей. Після цього виконано оновлення пакетів до актуальних версій. 

Таким чином забезпечується не тільки якість, а й безпека програмного продукту. 

  

ЛАБОРАТОРНА №8 Фінальний Quality Gate і Clean Project 

Метою восьмої лабораторної роботи було досягнення фінального стану «чистого» проєкту, який проходить усі перевірки Quality Gate. 

На цьому етапі проєкт уже містить: 

* автоматичний аналіз коду, 

* модульні тести, 

* контроль дублювання, 

* архітектурні перевірки, 

* оновлені залежності. 

Quality Gate визначає, що проєкт не має критичних багів, вразливостей і нових проблем якості. Якщо всі умови виконано — проєкт вважається готовим до злиття змін. 

Лабораторна №8 підсумовує весь курс і демонструє повний цикл реінжинірингу програмного забезпечення.
